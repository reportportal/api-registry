# AGENTS.md

This file provides guidance to coding agents working in this repository. It mirrors `CLAUDE.md` and applies to any agent (Claude Code, Codex, Cursor, Copilot, etc.).

## Repository purpose

API definitions registry for ReportPortal. Contains two parallel sources of truth:

- `api/openapi/` — OpenAPI specs (the main, actively-developed surface). Split across per-domain files in `api/openapi/apis/` (activities, groups, organizations, plugins, projects, users) that share reusable fragments in `enums/`, `models/`, `parameters/`, `types/`. The combined `api/openapi/reportportal-api.yaml` is **autogenerated** by `make join` — never edit it directly.
- `api/proto/` — gRPC/Protobuf definitions under `reportportal/<domain>/vN/`. Shared types live in `reportportal/common/`; new domains go under `reportportal/<area>/vN`.

The repo is migrating from code-first to API-first, so not every ReportPortal API is represented here yet. Generated client stubs land in `src/build/` and are gitignored unless a change explicitly requires updating them.

## Common commands

OpenAPI (via Redocly, invoked through `make`):

```bash
make lint                         # lint a single spec (defaults to reportportal-api.yaml)
make lint file=apis/users.yaml    # lint a specific file
make lint-all                     # lint every file under api/openapi/apis/
make join                         # rebuild api/openapi/reportportal-api.yaml from apis/*.yaml
make bundle                       # bundle one spec into build/openapi/
make bundle-all                   # dereferenced bundle of all apis/*.yaml
```

After editing shared fragments or any `apis/*.yaml`, run `make join` then `make bundle` to keep the combined spec and distributable bundles in sync.

Protobuf (via `buf`):

```bash
buf lint                                         # style/convention checks
buf breaking --against '.git#branch=main'        # compatibility check — run before pushing
buf build --exclude-source-info -o -#format=json | jq '.file[] | .package'   # verify compilation
buf generate                                     # remote-plugin codegen (Python, C#, JS, TS)
buf generate --template templates/buf.gen.opt.yaml     # optional languages (Go, Ruby, Kotlin, …)
buf generate --template templates/buf.gen.local.yaml   # local protoc-based generation
```

Java stubs use the Gradle protobuf plugin:

```bash
./gradlew generateProto
./gradlew clean
```

## API design guidelines

When designing or modifying OpenAPI specs, follow the project's own guidelines:

- [`docs/api-guidelines.md`](docs/api-guidelines.md) — concrete OpenAPI conventions for this repo. These are based on the Zalando [RESTful API and Event Scheme Guidelines](https://opensource.zalando.com/restful-api-guidelines).
- [`docs/api-principles.md`](docs/api-principles.md) — higher-level API design principles.
- [`docs/api-errors.md`](docs/api-errors.md) — standard REST error status codes and when to document each one on an endpoint.

Consult all three before adding new endpoints, models, or parameters.

## Conventions

- Protobuf follows the official Google style guide: `UpperCamelCase` messages, `snake_case` fields, enum values prefixed with a shared namespace (e.g. `LOG_TYPE_*`).
- OpenAPI YAML uses two-space indentation; files under `api/openapi/apis` use kebab-case names. Keep component IDs stable and reuse existing schemas from `models/`/`types/`/`enums/` rather than duplicating.
- Commit messages are short imperative statements without a trailing period (e.g. `Update log type schemas`). Avoid mixing proto and OpenAPI changes in one commit without explaining why.
- Treat `buf lint` + `buf breaking` + `buf build` as the pre-commit suite for proto changes, and `make lint` (or `lint-all`) for OpenAPI changes.
